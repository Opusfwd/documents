git提交规范

实施规范​

  • presit -> sit -> demo 冲突时以下游分支解决冲突
  • hotfix分支必须往下merged， 遗漏merge导致hotfix丢失个人负责
  • rebase, revert，解决别人冲突时需通知到群

场景​

  • 上迭代版本
    • presit -> sit-jp
    • presit -> sit-th
    • sit-jp -> demo-jp
  • hotfix冲突
    • demo -> sit-jp
    • sit-jp -> presit
    • sit-th -> presit
  • 上feature故事
    • feature -> presit
    • feature -> sit
    • feature -> demo
  • 上hotfix修复
    • hotfix -> presit
    • hotfix -> sit
    • hotfix -> demo
  • dit测试
    • hofitx -> develop
    • feature -> develop
  • 特殊
    • presit -> develop

注意点​

不能个人由下往上merge大分支​

  • 例如:（ presit -> sit ) ( sit -> demo )
  • 原因： presit -> sit -> demo 有一个迭代版本差异, 影响BA测试
  • 应由willie统一上版本，由demo -> UAT sit -> demo presit -> sit

分支冲突规避​

  • 单独合并（合除远程外的下级分支 cheery-pick )
    • hotfix分支合完对应分支后，cherry-pick自己的commit去合presit及以下分支
    • 优点： merge只有自己的修改，不会携带别人遗漏的hotfix， 冲突一目了然
    • 缺点： cherry-pick繁琐, 分支管理麻烦
  • 批量合并 (合除远程外的下级分支 整个分支合并 )
    • hotfix分支合完对应分支后，再将hotfix分支合回presit及以下分支
    • 优点： 不需要挑出自己的提交信息
    • 缺点： 可能携带别人的commit，甚至冲突，需灵活解决冲突 (自己或找人解决)

冲突场景​

  • 远程 -> 远程
    • presit -> sit 以presit为准
    • sit -> demo 以sit为准
    • sit -> presit 以presit为准
  • 分支 -> 远程 （hotfix/feature) (他人文件冲突以远程为准，自己冲突文件自己取舍)
    • hotfix/feature -> sit 以sit为准

rebase, revert，解决别人冲突时需通知到群​

  • 防止冲突误解决、代码覆盖

检出的分支不能往上游分支合并，跨迭代(remote)合并​

  • feature分支在迭代阶段要上sit及以上分支，需通知luke （确保分支干净）
    • cherry-pick自己的commit
    • 检出干净分支拷贝自己的更改文件上去

上迭代版本前需从上往下分支合并​

  • 原因： 避免遗漏hotfix忘记合回下游分支，解决冲突
  • 操作
    • demo -> sit-jp -> presit
    • sit-th -> presit
